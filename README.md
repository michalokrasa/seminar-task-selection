# Task Selection — CodeContests → Gherkin

Pipeline for picking **15 programming problems (5 easy / 5 medium / 5 hard)** from
[`open-thoughts/CodeContests`](https://huggingface.co/datasets/open-thoughts/CodeContests)
that are well-suited for translating their requirements into **Gherkin** (BDD
`Given/When/Then` scenarios).

## Goal & constraints

The downstream task is **manual translation of problem requirements to Gherkin**,
so candidates must be:

- **Behavior-describable** — discrete inputs/outputs and rule-based logic, not
  numerical/asymptotic reasoning.
- **Low-math** — no modular arithmetic, number theory, probability, geometry.
- **Self-contained** — no images, no references to attached figures.
- **Right-sized** — long enough to be interesting, short enough to fit on a page.
- **Difficulty-stratified** — explicitly bucketed easy / medium / hard.

## Why filtering is needed

The dataset has **9,644 tasks** packed as gzipped tar archives inside a parquet
file. Each archive contains an `instruction.md` plus test data. Two important
observations from prior investigation:

1. The `task.toml` `difficulty = "medium"` field is **hardcoded boilerplate** —
   it is the same string for every task and carries no signal.
2. The only meaningful difficulty signal is the **Codeforces Rating** preserved
   inside `instruction.md` under the `## Contest Information` block. Many tasks
   come from non-Codeforces sources and have `Rating: 0` (= unknown).

So the filtering must (a) parse `instruction.md`, (b) drop unrated tasks, and
(c) apply tag-based and keyword-based heuristics to remove math-heavy problems.

## Pipeline

```
tasks.parquet ─► extract instruction.md ─► parse Contest Info
              ─► filter (rating, tags, math, length, no <image>)
              ─► bucket by rating
              ─► random shuffle, sort by math_hits, take top N
              ─► write candidates_{easy,medium,hard}.md + index.csv
```

### 1. Difficulty buckets (Codeforces Rating)

| Bucket | Rating range | Codeforces equivalent |
| ------ | ------------ | --------------------- |
| easy   | 800 – 1100   | Div2 A / easy Div2 B  |
| medium | 1300 – 1600  | Div2 C / Div3 D-E     |
| hard   | 1800 – 2200  | Div2 D-E / Div1 B-C   |

Tasks with `Rating == 0` (no Codeforces source) and tasks outside these bands
are discarded. The gaps (e.g. 1200, 1700) are intentional — they keep the
buckets visibly distinct rather than fuzzy.

### 2. Tag allow / block lists

A task is kept iff its tag set intersects **ALLOWED_TAGS** _and_ does not
intersect **FORBIDDEN_TAGS**. See `select_tasks.py` for the full lists.

- **Allowed**: `implementation`, `strings`, `greedy`, `brute force`, `sortings`,
  `constructive algorithms`, `two pointers`, `*special`, `data structures`.
- **Forbidden**: `math`, `number theory`, `combinatorics`, `probabilities`,
  `geometry`, `dp`, `graphs`, `trees`, `dsu`, `flows`, `bitmasks`, `fft`,
  `matrices`, `divide and conquer`, `binary search`, `dfs and similar`,
  `string suffix structures`, `hashing`, `games`, `expression parsing`, ….

### 3. Math-iness keyword penalty

The `instruction.md` text is searched for a list of math keywords (`mod`,
`prime`, `gcd`, `factorial`, `permutation`, `probability`, `expected value`,
`10^9 + 7`, `998244353`, `binomial`, `fibonacci`, `matrix`, `polynomial`).
Any task with **more than 2** total hits is dropped, even if its tags pass.

### 4. Other heuristics

- Drop tasks containing `<image>` (figures lost during dataset prep → ambiguous).
- Drop tasks with `instruction.md` shorter than 400 chars or longer than 6000.

### 5. Sampling

Within each bucket, tasks are shuffled with a fixed seed (`RANDOM_SEED = 7`,
reproducible) and then **sorted ascending by math-keyword hits**, so the
cleanest, least-math-tinged candidates float to the top. Top
`PER_BUCKET = 12` are written.

> 12 candidates per bucket is intentional — you pick the final 5 by hand after
> reading them. This leaves slack for rejecting any that turn out poorly suited
> for Gherkin once you actually look at them.

## How to run (step by step)

All commands run from the repo root (`seminar/`).

### Setup (once)

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

### Step 1 — Download the dataset

```bash
bash scripts/1_download.sh
```

Downloads `data/tasks.parquet` (~42 MB) from HuggingFace. Skipped if already present.

### Step 2 — Filter & bucket candidates

```bash
.venv/bin/python scripts/2_select_tasks.py
```

Reads `data/tasks.parquet`, applies tag/math filters, and writes:

- `data/candidates/candidates_{easy,medium,hard}.md` — 12 candidates per bucket
- `data/index.csv` — flat overview of all 36 candidates

Run takes ~30 seconds. Outputs are deterministic (fixed seed).

### Step 3 — Manual task selection

Open `data/candidates/candidates_easy.md`, read all 12 tasks, and pick 5. Repeat
for medium and hard. Record your choices in `data/selection.csv` (columns: `bucket`,
`path`, `name`, `rating`, `tags`, `math_hits`, `length`, `accepted`).

### Step 4 — Extract selected task statements

```bash
.venv/bin/python scripts/3_extract_selected.py
```

Reads `data/selection.csv` and writes `selected/<bucket>/<slug>/task.md` for each
chosen task.

### Step 5 — Write descriptions (manual)

For each task in `selected/`, write either or both:

- `gherkin/description.feature` — Gherkin BDD scenarios
- `nl/description.md` — plain-language description

### Step 6 — Generate solutions

```bash
export OPENAI_API_KEY="sk-..."
.venv/bin/python scripts/4_generate_solutions.py --format nl
# or
.venv/bin/python scripts/4_generate_solutions.py --format gherkin
```

Writes `solution.py` alongside each description file.

### Step 7 — Run tests

Start SandboxFusion first (see [SandboxFusion Setup](#sandboxfusion-setup)), then:

```bash
.venv/bin/python scripts/5_run_tests.py
```

## Final selection (manual step)

After running the pipeline:

1. Open `candidates/candidates_easy.md`, read all 12 tasks.
2. Pick 5 that read most naturally as scenarios (clear actor, clear actions,
   small enumerable cases).
3. Repeat for `candidates_medium.md` and `candidates_hard.md`.
4. Record the chosen `path` IDs (e.g. `code_contests-0153`) somewhere stable so
   the selection is reproducible regardless of dataset re-shuffles.

### Tips for the manual pick

- Prefer problems whose statement names a concrete domain object (a juicer, a
  computer, a chat server, a flag) — they map naturally to Gherkin nouns.
- Prefer problems whose examples list 2–3 small distinct cases — each becomes
  one `Scenario`.
- Skip problems whose I/O is "T test cases, each line is …" only if you don't
  want to model batch behaviour; they're fine if you treat each test case as
  a Scenario Outline row.

## Code Generation & Testing Pipeline

After the Gherkin descriptions are created, the next step is:

1. **Generate code** — use an LLM to produce a `solution.py` for each task, guided by the
   `description.feature` (Gherkin) or `task.md` prompt. Place the generated file at:
   ```
   selected/<difficulty>/<task-slug>/solution.py
   ```
2. **Run tests** — validate every `solution.py` via `scripts/5_run_tests.py`, which:
   - Pulls `tests/test_data.json` (10 judge test cases) and `tests/test_state.py` (pytest
     harness) directly out of each task's tarball in `data/tasks.parquet`.
   - Sends them to SandboxFusion `/run_code` with `language=pytest`, injecting
     `solution.py` and `test_data.json` as sandbox files.
   - Reports pass/fail based on the pytest exit code.

   The real test suite (10 cases per task) is **richer than the 1–4 sample examples** shown
   in the problem statement — it is the authoritative judge data from the original dataset.

### Generating solutions with GPT

`scripts/4_generate_solutions.py` calls the OpenAI API with the chosen description file as the
prompt and writes the response to `solution.py` in the same subfolder.

**Prerequisites**: set the `OPENAI_API_KEY` environment variable.

```bash
export OPENAI_API_KEY="sk-..."
```

#### Options

| Flag        | Required | Choices           | Description                                                                             |
| ----------- | -------- | ----------------- | --------------------------------------------------------------------------------------- |
| `--format`  | yes      | `gherkin` \| `nl` | Description file to use. Falls back to the other format if the preferred one is absent. |
| `--problem` | no       | path              | Path to a single problem folder. Omit to process every problem under `selected/`.       |

The output file mirrors the chosen format's subfolder:

- `--format nl` → `<problem>/nl/solution.py`
- `--format gherkin` → `<problem>/gherkin/solution.py`

Existing `solution.py` files are always overwritten.

#### Examples

```bash
# Generate solutions for all problems using NL descriptions
.venv/bin/python scripts/4_generate_solutions.py --format nl

# Generate a solution for a single problem using Gherkin
.venv/bin/python scripts/4_generate_solutions.py --format gherkin \
  --problem selected/easy/5-a-chat-servers-outgoing-traffic
```

### SandboxFusion Setup

SandboxFusion is a Docker-based code execution sandbox. It runs generated Python code in
isolation and returns stdout/stderr.

**Prerequisites**: Docker Desktop (or Docker Engine) installed and running.

#### 1. Pull & start the sandbox

```bash
docker run -it -p 8080:8080 volcengine/sandbox-fusion:server-20250609
```

The sandbox listens on `http://localhost:8080`. Verify it is healthy:

```bash
curl http://localhost:8080/run_code \
  -H 'Content-Type: application/json' \
  --data-raw '{"code": "print(\"ok\")", "language": "python"}'
```

Expected output contains `"stdout": "ok\n"`.

#### 2. Install dependencies

From the **repo root** (`seminar/`), create a venv and install all dependencies at once:

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

### Running the tests

From the **repo root** (`seminar/`):

```bash
.venv/bin/python scripts/5_run_tests.py
```

#### Options

| Flag                                   | Default                 | Description                                       |
| -------------------------------------- | ----------------------- | ------------------------------------------------- |
| `--sandbox URL`                        | `http://localhost:8080` | SandboxFusion base URL                            |
| `--timeout N`                          | `60`                    | Per-task pytest run timeout (seconds)             |
| `--verbose`                            | off                     | Print full pytest stdout on failure               |
| `--difficulty easy\|medium\|hard\|all` | `all`                   | Limit to one difficulty bucket                    |
| `--task SUBSTRING`                     | all tasks               | Only tasks whose folder name contains `SUBSTRING` |

#### Examples

```bash
# All commands run from seminar/ (the repo root)

# Test only easy tasks, with verbose pytest output
.venv/bin/python scripts/5_run_tests.py --difficulty easy --verbose

# Test a single task
.venv/bin/python scripts/5_run_tests.py --task 5-a-chat-servers-outgoing-traffic --verbose

# Use a remote sandbox
.venv/bin/python scripts/5_run_tests.py --sandbox http://192.168.1.10:8080
```

#### Expected output

```
Loading selection index and parquet …
  9644 tasks in parquet, 21 in selection.csv

[PASS] easy/1173-a-nauuo-and-votes
[PASS] easy/1281-a-suffix-three
...
============================================================
Tasks : 21/21 passed
All tasks passed.
```

Tasks without a `solution.py` are silently skipped (`[SKIP]`), so you can run the script
incrementally as solutions are generated.

## Tuning knobs

If the candidate pool feels off, edit constants at the top of `scripts/2_select_tasks.py`:

- `BUCKETS` — change the rating ranges.
- `ALLOWED_TAGS` / `FORBIDDEN_TAGS` — broaden/narrow tag policy.
- `MATH_KEYWORDS` / `MATH_PENALTY_THRESHOLD` — math sensitivity.
- `PER_BUCKET` — how many candidates surface per bucket.
- `RANDOM_SEED` — change to get a different sample.
